# Standards & API Reference

> Project: Cloud Security Posture Management · Generated: 2026-05-03

## Industry Standards & Specifications

### ISO Standards

**ISO/IEC 27001:2022 — Information Security Management Systems**
- URL: https://www.iso.org/standard/27001
- Relevance: Annex A Controls 8.7 (protection against malware), 8.9 (configuration management), and 8.25 (secure development lifecycle) are directly addressed by CSPM. Annex A 8.9 requires that configurations of hardware, software, and networks are established, documented, implemented, monitored, and reviewed — precisely what a CSPM automates. ISO 27001 certification audit evidence (configuration monitoring, access control compliance, change management) is substantially generated by CSPM continuous monitoring.

**ISO/IEC 27017:2015 — Code of Practice for Information Security Controls Based on ISO/IEC 27002 for Cloud Services**
- URL: https://www.iso.org/standard/43757.html
- Relevance: Cloud-specific extension of ISO 27002 providing implementation guidance for cloud customers and cloud service providers. Defines cloud-specific controls for shared responsibility, virtual machine hardening, cloud administrator access, and segregation in virtual computing environments — all directly mapped to CSPM detection rules. CSPM compliance reports for cloud deployments should reference ISO 27017 controls alongside ISO 27001.

### W3C & IETF Standards

**RFC 6749 — OAuth 2.0 Authorization Framework**
- URL: https://datatracker.ietf.org/doc/html/rfc6749
- Relevance: CSPM platforms must use OAuth 2.0 to authenticate with cloud provider APIs (AWS, Azure, GCP) for agentless scanning. Cloud provider SDKs (AWS SDK, Azure SDK, Google Cloud SDK) all use OAuth 2.0 / cloud-native credential flows for programmatic access. CSPM tools must implement least-privilege OAuth scopes — read-only discovery permissions — to minimise the risk of compromise.

**RFC 7231 — HTTP/1.1 Semantics and Content**
- URL: https://datatracker.ietf.org/doc/html/rfc7231
- Relevance: Defines HTTP method semantics and status codes used by all CSPM REST APIs (AWS Security Hub, Azure Defender for Cloud, GCP Security Command Center, Wiz, Orca). Correct rate-limit handling (HTTP 429) and pagination patterns are essential for large-scale CSPM integrations scanning millions of cloud resources.

**RFC 7807 — Problem Details for HTTP APIs**
- URL: https://datatracker.ietf.org/doc/html/rfc7807
- Relevance: Standardises structured error payloads from REST APIs. CSPM platforms should adopt RFC 7807 for their own APIs to ensure integration partners receive machine-parseable errors for authentication failures, scan permission errors, and resource-not-found responses.

### Data Model & API Specifications

**OCSF — Open Cybersecurity Schema Framework**
- URL: https://ocsf.io/ ; https://github.com/ocsf/ocsf-schema
- Relevance: OCSF is an open, vendor-neutral standard for cybersecurity event logging and data normalisation, co-developed by AWS, Splunk, CrowdStrike, and 15+ security vendors. Now a Linux Foundation project (November 2024). CSPM finding events (misconfiguration detected, configuration changed, resource exposed) should be expressed in OCSF format to enable interoperability with SIEM, SOAR, and data lake ingestion pipelines. Orca Security implemented OCSF support in AWS Security Hub integration (June 2025). CSPM platforms targeting enterprise buyers must emit OCSF-formatted events.

**OSCAL — Open Security Controls Assessment Language (NIST)**
- URL: https://pages.nist.gov/OSCAL/ ; https://github.com/usnistgov/OSCAL
- Relevance: NIST-developed machine-readable language (XML, JSON, YAML) for expressing security control catalogs, baselines, system security plans, and assessment results. NIST SP 800-53 Rev 5, CSF v2.0, NIST SP 800-171, and FedRAMP baselines are all published as OSCAL catalogs. CSPM compliance reporting for US government (FedRAMP, FISMA) must output findings in OSCAL format to interoperate with GRC tools. OSCAL reduces audit duration from months to minutes by providing machine-readable compliance artefacts.

**OpenAPI Specification 3.1 (OAS 3.1)**
- URL: https://spec.openapis.org/oas/v3.1.0.html ; https://swagger.io/specification/
- Relevance: The industry standard for REST API documentation. AWS Security Hub, Microsoft Defender for Cloud, and Orca Security all publish OpenAPI/Swagger specifications. A CSPM platform must publish a comprehensive OAS 3.1 specification for its findings API, asset inventory API, compliance report API, and webhook event schema — enabling integration partners to auto-generate SDKs and write contract tests.

**JSON Schema 2020-12**
- URL: https://json-schema.org/specification
- Relevance: Standard for validating JSON payloads. CSPM finding payloads (resource ID, misconfiguration type, severity, compliance framework mapping, remediation steps) and IaC scan results must be validated against published JSON Schemas. Critical for ensuring data quality in SIEM ingestion and compliance evidence pipelines.

**GraphQL — SDL and Query Language**
- URL: https://graphql.org/learn/schema/ ; https://spec.graphql.org/
- Relevance: Wiz and Orca Security expose GraphQL APIs for flexible querying of CSPM findings, asset relationships, and attack paths. GraphQL is particularly well-suited to CSPM use cases because security analysts need to compose complex queries joining findings, resources, identities, and network exposure — graph-structured data that maps naturally to GraphQL's hierarchical schema. CSPM platforms with attack path analysis should evaluate GraphQL alongside REST for their query APIs.

### Security & Compliance Frameworks (De Facto Standards)

**CIS Benchmarks — Cloud Security Configuration Baselines**
- URL: https://www.cisecurity.org/cis-benchmarks ; https://learn.cisecurity.org/benchmarks
- Relevance: The most widely referenced technical hardening standard in CSPM rule sets. CIS publishes prescriptive configuration baselines for AWS (107 recommendations), Azure (195 recommendations), and GCP (133 recommendations) organised into Level 1 (baseline security) and Level 2 (defence in depth). All CSPM platforms ship detection rules mapped to CIS Benchmarks. AWS Security Hub supports CIS AWS Foundations Benchmark versions 5.0.0, 3.0.0, 1.4.0, and 1.2.0. CSPM platforms must support the current CIS Benchmark versions and update rules when new versions are published.

**NIST SP 800-53 Rev 5 — Security and Privacy Controls**
- URL: https://csrc.nist.gov/pubs/sp/800/53/r5/upd1/final
- Relevance: The US federal security control catalogue for information systems. CSPM finding-to-control mapping enables FedRAMP and FISMA compliance evidence generation. The controls are available in OSCAL format (JSON, XML, YAML) for machine-readable integration. Both AWS Security Hub and Azure Policy provide native NIST SP 800-53 Rev 5 mappings for their CSPM rules. CSPM platforms targeting US government buyers must map all findings to NIST 800-53 controls.

**NIST Cybersecurity Framework 2.0 (CSF 2.0)**
- URL: https://www.nist.gov/cyberframework
- Relevance: NIST CSF 2.0 defines six functions (Govern, Identify, Protect, Detect, Respond, Recover). CSPM primarily implements Identify (asset inventory, configuration assessment) and Detect (continuous monitoring, misconfiguration alerting). CSF 2.0 is available as an OSCAL catalog. CSPM compliance dashboards should support CSF 2.0 mapping alongside CIS and NIST 800-53.

**PCI DSS v4.0 — Requirement 6 and Requirement 12**
- URL: https://www.pcisecuritystandards.org/document_library/
- Relevance: PCI DSS Requirement 6 (Develop and Maintain Secure Systems) and Requirement 12 (Support Information Security with Organisational Policies) mandate configuration management and security baseline monitoring for cloud environments handling cardholder data. CSPM continuous monitoring automates PCI DSS Requirement 6.3.3 (patch management evidence), 6.4.1 (public-facing application security), and 12.3 (configuration baseline documentation). CSPM compliance reports must support PCI DSS v4.0 control mapping.

**SOC 2 — Trust Services Criteria (AICPA)**
- URL: https://www.aicpa-cima.com/resources/landing/system-and-organization-controls-soc-suite-of-services
- Relevance: CSPM continuous monitoring provides substantial evidence for SOC 2 Type II audits. The Availability and Security availability criteria require continuous infrastructure monitoring; CSPM monitoring logs, configuration snapshots, and change alerts serve as direct SOC 2 evidence artefacts. Well-designed CSPM platforms can reduce SOC 2 evidence collection time from 200–400 hours to 20–40 hours per audit cycle.

### Security & Authentication Standards

**OWASP API Security Top 10 (2023)**
- URL: https://owasp.org/API-Security/editions/2023/en/0x11-t10/
- Relevance: CSPM APIs that expose cloud inventory, misconfiguration findings, and compliance evidence are high-value targets. Key risks: Broken Object Level Authorization (a CSPM user in account A must not access findings from account B); Broken Authentication (cloud provider API credentials used for scanning must be read-only and audited); Unrestricted Resource Consumption (CSPM bulk export APIs can retrieve millions of findings requiring pagination and rate limiting); and Security Misconfiguration (CSPM management APIs must not be publicly accessible).

**GDPR — General Data Protection Regulation (EU 2016/679)**
- URL: https://gdpr.eu/
- Relevance: CSPM platforms performing data security posture management (DSPM) — scanning cloud storage, databases, and data lakes to detect PII/PHI — process personal data under GDPR. The act of identifying and storing the location of personal data in cloud resources is itself a processing activity. CSPM platforms must implement data minimisation (storing only metadata, not content), purpose limitation (using scan results only for security purposes), and appropriate Data Processing Agreements with all cloud provider APIs used for scanning.

### MCP Server Specifications

**Model Context Protocol (MCP) — Agentic CSPM and Prowler AI Integration**
- URL: https://modelcontextprotocol.io/ ; https://docs.prowler.com/
- Relevance: MCP is the emerging standard for connecting AI agents to security tools. Prowler — the leading open-source CSPM platform — has implemented a Prowler MCP Server for its "Lighthouse" AI security assistant, enabling AI agents to query CSPM findings, compliance status, and remediation guidance via natural language tool calls. This is a significant early signal: CSPM platforms that expose MCP servers enable AI agents to autonomously answer questions like "which production accounts fail PCI DSS Requirement 6?" or "generate a remediation plan for the top 10 findings affecting the payments service." AI-SPM (AI Security Posture Management) is an emerging CSPM sub-category tracking AI model deployments, MCP servers, and agent workloads as new cloud resource types requiring posture monitoring.

## Notes

**ASFF (AWS Security Finding Format) as a De Facto Standard:** AWS Security Hub's finding format (ASFF) has become a de facto industry format for CSPM findings ingested into AWS Security Hub from third-party tools. Many CSPM vendors (Orca, Prowler, Lacework) implement ASFF output to integrate with the AWS security ecosystem. A CSPM platform targeting multi-cloud should emit both OCSF-formatted events and ASFF for AWS Security Hub integration.

**OSCAL Adoption Trajectory:** OSCAL is mandatory for FedRAMP compliance automation and is being adopted by GRC tools (Regscale, Isora) for machine-readable compliance evidence. CSPM platforms targeting the US government market must output OSCAL-formatted assessment results. OSCAL eliminates the need for spreadsheet-based compliance evidence and can reduce audit duration from months to minutes.

**Prowler MCP Server as Reference Implementation:** Prowler's MCP Server for the "Lighthouse" AI assistant represents the first open-source CSPM + MCP integration. It provides a reference implementation for how AI agents can query CSPM findings, compliance status, and remediation steps via natural language tool calls. CSPM platforms should evaluate Prowler's MCP Server design before implementing their own.

**AI-SPM as Emerging CSPM Sub-Category:** AI Security Posture Management (AI-SPM) is an emerging discipline tracking AI models, training datasets, MCP servers, and agentic workloads as new cloud resource types requiring posture monitoring. Wiz introduced AI-SPM features in 2024 (Shadow AI detection). As MCP server deployments grow (10,000+ public MCP servers by Q2 2026), CSPM platforms must extend their asset inventories and misconfiguration rules to cover AI/ML infrastructure alongside traditional cloud resources.
